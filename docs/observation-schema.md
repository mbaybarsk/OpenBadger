# OpenBadger Observation Schema  
  
## Purpose  
  
OpenBadger scanners and sensors do **not** write directly to asset records.  
  
They emit **observations**.  
  
An observation is an immutable, normalized piece of evidence about a device, address, interface, or network relationship. The server stores observations, then correlates them into assets and sightings.  
  
This document defines the v0.1 observation format used by:  
  
- active collectors  
- passive sensors  
- the server-side correlation engine  
  
## Key Principles  
  
1. **Observations are immutable**  
   - New evidence creates new observations.  
   - Existing observations are not edited in place.  
  
2. **Observations are site-aware**  
   - Identity and correlation are always evaluated within a site context.  
  
3. **Scanners report evidence, not final truth**  
   - A single protocol may only know part of a device's identity.  
   - The server decides how observations become assets.  
  
4. **No secrets in observations**  
   - Observations must never contain plaintext passwords, private keys, SNMP communities, or token material.  
  
5. **No packet payload storage in v0.1**  
   - Passive observations contain metadata only.  
  
6. **Passive observations must be aggregated**  
   - Sensors must not emit one observation per packet or one observation per raw flow record.  
  
## Observation Categories  
  
Each observation has a `scope` value:  
  
- `asset`  
  - facts directly describing a device  
  
- `sighting`  
  - evidence that a device or address was seen at a point or during an interval  
  
- `relationship`  
  - evidence linking one thing to another  
  - examples:  
    - IP-to-MAC mappings  
    - switch FDB entries  
    - future LLDP/CDP neighbor relationships  
  
## Common Envelope  
  
All observations use the same top-level envelope.  
  
| Field | Type | Required | Description |  
|---|---|---:|---|  
| `schema_version` | string | yes | Observation schema version. v0.1 uses `0.1`. |  
| `observation_id` | string | yes | UUID generated by the emitter. |  
| `type` | string | yes | Observation type, for example `snmp.system`. |  
| `scope` | string | yes | One of `asset`, `sighting`, `relationship`. |  
| `site_id` | string | yes | Site context for identity and correlation. |  
| `job_id` | string | no | Present for scheduled active scans. |  
| `emitter` | object | yes | Identity of the collector or sensor that emitted the observation. |  
| `observed_at` | string | yes | RFC3339 UTC timestamp for the observation. |  
| `target` | object | no | Active scan target context. |  
| `identifiers` | object | no | Stable or semi-stable identifiers. |  
| `addresses` | object | no | Addressing and interface evidence. |  
| `facts` | object | yes | Type-specific normalized facts. |  
| `relations` | array | no | Type-specific relationship details. |  
| `evidence` | object | yes | Confidence and source metadata. |  
| `raw` | object | no | Sanitized protocol-specific details. |

## Object Definitions  
  
### `emitter`  
  
Describes the node that created the observation.  
  
| Field | Type | Required | Description |  
|---|---|---:|---|  
| `kind` | string | yes | `collector` or `sensor` |  
| `id` | string | yes | Node ID |  
| `name` | string | yes | Human-readable node name |  
| `version` | string | yes | OpenBadger version |  
| `capability` | string | yes | Capability that produced the observation, such as `snmp`, `ssh`, `winrm`, `pcap`, `flow` |  
  
Example:  
  
```json  
{  
  "kind": "sensor",  
  "id": "sensor-site-a-01",  
  "name": "site-a-sensor-01",  
  "version": "0.1.0",  
  "capability": "pcap"  
}  
```  
  
### `target`  
  
Present mainly for active scans.  
  
| Field | Type | Required | Description |  
|---|---|---:|---|  
| `input` | string | no | Original scheduled target, such as an IP, hostname, or CIDR-derived host |  
| `ip` | string | no | Resolved target IP |  
| `hostname` | string | no | Original hostname used for the scan, if applicable |  
| `protocol` | string | no | `icmp`, `snmp`, `ssh`, or `winrm` |  
| `port` | number | no | Protocol port where applicable |  
  
### `identifiers`  
  
Identifiers are used by correlation. Include as many as are available and trustworthy.  
  
| Field | Type | Required | Description |  
|---|---|---:|---|  
| `hostnames` | array of strings | no | Lower-case normalized hostnames |  
| `fqdn` | string | no | Lower-case normalized FQDN |  
| `mac_addresses` | array of strings | no | Canonical lower-case colon-separated MACs |  
| `serial_number` | string | no | Device serial number |  
| `system_uuid` | string | no | Canonical system UUID |  
| `bios_uuid` | string | no | BIOS/system UUID, often useful for Windows |  
| `snmp_engine_id` | string | no | Strong identifier for many SNMP devices |  
| `ssh_host_key_fingerprints` | array of strings | no | SSH host key fingerprints, such as `SHA256:...` |  
| `machine_id` | string | no | Linux machine ID where safely available |  
  
### `addresses`  
  
Addressing and interface context.  
  
| Field | Type | Required | Description |  
|---|---|---:|---|  
| `ip_addresses` | array of strings | no | IPv4 or IPv6 addresses |  
| `vlan_ids` | array of numbers | no | VLAN IDs if observed |  
| `subnet_cidrs` | array of strings | no | Optional network context |  
| `interface_name` | string | no | Interface name related to this observation |  
| `interface_index` | number | no | SNMP ifIndex or similar |  
  
### `evidence`  
  
Metadata about confidence and source context.  
  
| Field | Type | Required | Description |  
|---|---|---:|---|  
| `confidence` | number | yes | Range `0.0` to `1.0` |  
| `source_protocol` | string | yes | Protocol that produced the observation |  
| `credential_profile` | string | no | Name or ID of the credential profile used |  
| `first_seen` | string | no | RFC3339 UTC, mainly for passive summaries |  
| `last_seen` | string | no | RFC3339 UTC, mainly for passive summaries |  
| `packet_count` | number | no | Passive packet metadata count |  
| `byte_count` | number | no | Flow or passive byte count |  
| `flow_count` | number | no | Aggregated flow count |  
  
### `raw`  
  
Optional sanitized protocol-specific data.  
  
Rules:  
  
- may be omitted entirely  
- must not contain secrets  
- must not contain packet payloads  
- should be truncated if excessively large  
  
---  
  
## Normalization Rules  
  
### Timestamps  
  
- Use UTC  
- Use RFC3339 format  
  
Example:  
  
```text  
2026-03-31T12:15:00Z  
```  
  
### Hostnames  
  
- Store normalized lower-case values in `identifiers.hostnames` and `identifiers.fqdn`  
- Preserve original casing only in `raw` if needed  
  
### MAC addresses  
  
Use lower-case colon-separated format.  
  
Example:  
  
```text  
00:11:22:33:44:55  
```  
  
### UUIDs  
  
Use canonical string form.  
  
Example:  
  
```text  
7d9c1a4d-3f4f-4f0b-9159-9e4f7acb2d83  
```  
  
### IP addresses  
  
- Store only literal IPs in `addresses.ip_addresses`  
- Do not place hostnames in IP fields  
  
### Confidence Guidance  
  
| Range | Meaning |  
|---|---|  
| `0.95 - 1.00` | Strong identity evidence |  
| `0.70 - 0.94` | Good but not absolute evidence |  
| `0.30 - 0.69` | Weak or passive-only evidence |  
| `< 0.30` | Usually too weak to be useful in v0.1 |

## v0.1 Observation Types  
  
| Type | Scope | Purpose |  
|---|---|---|  
| `icmp.alive` | sighting | Host responded to ICMP |  
| `snmp.system` | asset | Core system identity from SNMP |  
| `snmp.interface` | asset | Interface detail from an SNMP device |  
| `snmp.arp_entry` | relationship | IP-to-MAC evidence learned from an SNMP device |  
| `snmp.fdb_entry` | relationship | Switch FDB/MAC table evidence |  
| `ssh.host` | asset | Linux/Unix host identity and OS facts |  
| `winrm.windows_host` | asset | Windows host identity and OS facts |  
| `passive.pcap_sighting` | sighting | Aggregated packet-metadata-based sighting |  
| `flow.sighting` | sighting | Aggregated flow-based sighting |

## Correlation Guidance for v0.1  
  
The correlation engine should prefer identifiers in this order.  
  
### Strong identifiers  
  
- `serial_number`  
- `system_uuid`  
- `bios_uuid`  
- `snmp_engine_id`  
- `ssh_host_key_fingerprints`  
  
### Medium identifiers  
  
- `mac_addresses`  
- `hostnames`  
- `fqdn`  
  
### Weak identifiers  
  
- `ip_addresses`  
- DHCP hostname hints  
- mDNS names  
- NBNS names  
- flow-only sightings  
  
### Rules  
  
1. Never use IP address alone as a durable global identity.  
2. Correlation must always be site-aware.  
3. Weak-only evidence may create a provisional asset.  
4. Strong identifiers may merge observations into one asset.  
5. Conflicting strong identifiers must not auto-merge.  
  
---  
  
## Storage Guidance  
  
For v0.1, the recommended implementation is:  
  
- store the full observation body as JSONB  
- extract and index important fields such as:  
  - `site_id`  
  - `type`  
  - `scope`  
  - `observed_at`  
  - selected identifiers  
  - selected IP and MAC values  
  
Suggested index candidates:  
  
- `site_id`  
- `type`  
- `observed_at`  
- GIN index on JSONB body  
- derived searchable columns for IP, MAC, serial, hostname, UUID  
  
---  
  
## Validation Rules  
  
At minimum, the server should reject observations that are missing:  
  
- `schema_version`  
- `observation_id`  
- `type`  
- `scope`  
- `site_id`  
- `emitter`  
- `observed_at`  
- `facts`  
- `evidence.confidence`  
- `evidence.source_protocol`  
  
The server should also reject observations that:  
  
- include malformed timestamps  
- use invalid confidence values outside `0.0` to `1.0`  
- contain plaintext secrets in known sensitive fields  
- contain packet payload data in passive observations

## Summary  
  
The OpenBadger observation model exists to keep collection simple and correlation flexible.  
  
Scanners and sensors should only answer:  
  
- what was seen  
- when it was seen  
- how it was seen  
- how confident the emitter is  
  
The server is responsible for turning that evidence into a durable asset inventory.
